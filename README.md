# 12306 余票查询 · 买长乘短

网页版余票查询：输入出发地 / 目的地、勾选车次类型和座位类型并查询。
当目的区段 **B→C 直达没票** 时，自动尝试同一车次的：

- **A→C**：提前一站（或多站）上车
- **B→D**：延后一站（或多站）下车
- **A→D**：两头都延伸

有结果就在页面上列出对应的延伸方案与余票数量（即「买长乘短」）。


## 主要功能

- **余票查询 + 买长乘短**：直达没票时自动找「提前上车 / 延后下车」的可购区段。
- **多日期查询**：填结束日期可一次查多天（最多 5 天）。
- **票价显示与筛选**：每个有票座位显示票价，可设价格上限、按价格/出发到达时间/耗时/余票排序。
  （票价接口对非登录会话限流较严，按需/渐进加载，偶有缺失可点「¥查价」补拉。）
- **浏览器端监控**：开着网页定时重查，有票弹浏览器通知 + 提示音，可自动弹开下单页。
- **☁️ 服务端托管监控**：监控任务跑在后端线程，**关掉网页也继续跑**，有票直接推微信；
  进程重启自动恢复 running 任务。需先配置微信推送。
- **微信通知**：PushPlus / Server酱 / 企业微信群机器人，推送与网页一致的「结果卡片」。
- **去 12306 下单 / 候补**：每趟车生成官方下单页深链，已登录则直接进登录态。

## 运行

```bash
# 首次：创建虚拟环境并安装依赖
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt

# 启动
.venv/bin/python app.py
```

打开浏览器访问 http://127.0.0.1:5001 （可用环境变量 `PORT` 改端口）

### 多用户：每人用自己的 12306 账号

本服务是**多用户**的：谁打开页面，谁用 12306 App 扫**自己**的码登录，抢票就下单到**他自己**的账号；每个浏览器各自独立、互不可见（看不到也删不掉别人的任务）。

- 登录态**只存在服务器内存、绝不落盘**；服务重启或长时间空闲后各自需重新登录。
- 不再使用 `login_session.json`（旧文件可删）。

### 扫码登录怎么工作（重要）

访客点「获取二维码」后，服务器会**在站主本机开一个真 Chrome**（可见窗口）打开 12306 登录页、取出二维码显示到访客网页上；访客用手机 12306 App 扫码确认 → 服务器抠出登录态灌进该会话 → **立即关闭该 Chrome**。之所以用真 Chrome：12306 扫码后换正式登录态的流程在纯 HTTP 请求里会被风控 302 打回（设备指纹算法易碎），真 Chrome 能可靠跑通（已实测验证）。

- **需要站主电脑开着**、装有 Google Chrome 与 Node。登录期间你屏幕会弹出 Chrome 窗口（登录完自动关）。
- **并发上限 3**：同时最多 3 人处于「扫码登录中」；满了提示「稍后再试」。已登录的人不占名额（登录完 Chrome 就关了）。端口段固定 9322–9324。
- 「打开官方登录页 / 导入登录态」仅**站主本机**可见（远程访客用扫码）。

### 任务的控制与自动停止

为避免「访客开了抢票任务后走人、无人能停」：

- **管理员可控全部**：你（站主）用**管理员令牌**（`.admin_token`，启动时打印；浏览器点右上角「🛠 管理员」填入）后，能看到并停止/删除**所有人**的抢票/监控任务。普通访客仍只看/控自己的。
- **退出登录即停**：访客点「退出登录」会立即停掉他名下还在跑的抢票任务。
- **空闲超时即停**：只要页面开着，前端每分钟发一次心跳保活；一旦**关掉网页/走人**，会话在 `LOGIN_IDLE_TTL`（默认 30 分钟）后被回收，其名下抢票/监控任务自动停止、登录态丢弃。
- 手动指定管理员令牌：`ADMIN_TOKEN='你的强随机串' .venv/bin/python app.py`

### ⚠️ 访问令牌（入场票）

令牌控制的是「**谁能进这台服务器**」——进来后各人仍登录各自的 12306 账号。服务会读取或生成 `.app_token`（仅本机可读、已 gitignore）；`start.sh` / `share.sh` 启动时打印出来，浏览器点右上角「🔑 令牌」填入同一串即可。

```bash
./start.sh     # 本机使用：会打印令牌
./share.sh     # 对外分享：连同公网网址一起打印令牌，发给对方
```

- 经 cloudflared / 端口转发对外暴露时：**必须**带令牌（`share.sh` 已强制设置），否则任何拿到公网地址的人都能进来占用你的服务器。
- 手动指定令牌：`APP_TOKEN='换成你自己的强随机串' .venv/bin/python app.py`

> ⚠️ **风控与信任提示**：所有访客的 12306 请求都从**你这台服务器的同一个 IP** 发出（主流抢票软件是各自在自己电脑跑）。集中到一台 IP 会让风控集中，某个访客被风控可能连累整台 IP；且访客登录态在你服务器内存里、运营者理论上可访问。**建议只分享给小范围信任的人。**

其它环境变量：`HOST`（默认 `127.0.0.1`）、`PORT`、`FLASK_DEBUG`（对外分享务必不要开）、`FLASK_SECRET`（会话签名密钥，不设则读/生成 `.flask_secret`）、`LOGIN_IDLE_TTL`（登录态空闲驱逐秒数，默认 `1800`）、`ORDER_MAX_PARALLEL_TICKS`（抢票任务同时查询轮数，默认 `2`）、`ORDER_MIN_GAP`（相邻下单全局最小间隔秒，默认 `3`）、`ORDER_ACCOUNT_GAP`（同一账号两次下单最小间隔秒，默认 `8`）。

> **下单串行闸门**：多人共用一个出口 IP 时，多账号同一瞬间冲下单接口最容易触发 12306 风控。已加全局串行闸门 + 全局/每账号最小间隔（带 ±30% 抖动），把下单动作错开。这**缓解**但不**根治** IP 集中问题——彻底解决需要每账号独立代理 IP 或各自在自己电脑跑。间隔越大越安全但抢票越慢，可用上面两个环境变量按需调（设 `0` 关闭对应间隔）。

> **站主抄送通知**：设置 `OWNER_NOTIFY_CHANNEL`（`pushplus`/`serverchan`/`wecom`）+ `OWNER_NOTIFY_TOKEN`（对应 token/webhook）后，**抢到票**和**监控发现有票**时，除了通知访客自己，也给站主抄送一份。凭证只存服务端、**绝不下发前端**（访客拿不到你的 webhook）。访客本人就是站主（同渠道+token）时自动去重不重复推。留空则不抄送。
> ```bash
> OWNER_NOTIFY_CHANNEL='wecom' OWNER_NOTIFY_TOKEN='你的企业微信webhook' .venv/bin/python app.py
> ```
敏感文件均已 gitignore 且 `0600` 落盘；推送 token 在装了 `cryptography` 时字段级加密。

## 服务端托管监控用法

1. 在页面填好出发/目的/日期、座位等条件，并在「微信通知」里配好渠道 + token（先点「测试」确认能收到）。
2. 点「☁️ 服务端托管监控」里的 **＋ 用当前条件创建托管任务**。
3. 之后可**关闭网页**，只要运行 `app.py` 的电脑别关机，后端就持续查、有票推微信。
4. 面板里可查看每个任务的轮次/上次检查/累计命中，并「查看最近结果 / 停止 / 删除」。

## 说明 / 注意

- 数据来自 12306 官网公开查票接口，仅供个人查询参考。
- 首次查询会先访问 12306 init 页面获取 Cookie，否则接口会返回错误页。
- 含延伸 / 票价的查询会发起多次请求，已内置限速闸门和请求上限，请避免频繁刷新以免被风控限速。
- 票价接口（queryTicketPrice）对非登录会话限流较严，无法一次拉全；票价会按需渐进加载、服务端缓存，缺失项可点「¥查价」重试。
- 托管任务的 token 仅保存在本地 `monitor_jobs.json`；任务越多每轮越慢，**监控间隔建议 ≥ 30 秒**。
- 「买长乘短」只在你的区段处于车次**中间段**时才可能有方案；若出发站是始发站、到达站是终到站，则两头无法再延伸（属正常情况）。
- 部分区段 12306 不单独售卖（停靠 ≠ 该区段可购票），这类延伸会自动跳过。
